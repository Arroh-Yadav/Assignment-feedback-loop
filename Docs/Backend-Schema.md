# Backend Schema — Assignment Feedback Loop
## IPS Academy, Indore

---

## Overview

The database has **14 tables** built on PostgreSQL (hosted on Supabase).
All IDs are UUIDs. Enums are used for role and status fields.

---

## Table Relationship Summary

```
USERS
  ├── STUDENTS → SECTIONS → SUB_BRANCHES → BRANCHES
  └── FACULTY  → BRANCHES
                    └── FACULTY_SECTIONS → SECTIONS
  └── NOTIFICATIONS (per user)

ASSIGNMENTS (created by FACULTY for a SECTION)
  └── SUBMISSIONS (by STUDENTS)
        ├── AI_FEEDBACK (auto-generated by Gemini)
        └── EVALUATIONS (by FACULTY)
              └── ANNOTATIONS

SYNC_LOG (standalone — admin-triggered institute DB comparison runs)
```

## 1. `users` — Master auth table

Every person on the platform has one row here regardless of role.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `role` | ENUM (`student`, `faculty`, `admin`) | User type |
| `employee_or_computer_code` | VARCHAR | Login identity |
| `enrollment_or_email` | VARCHAR | Second login credential |
| `is_active` | BOOLEAN | Synced from institute DB |
| `created_at` | TIMESTAMP | First login time |
| `last_login` | TIMESTAMP | Session tracking |

## 10. `ai_feedback` — Gemini's analysis result

One row per submission, generated automatically after upload.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `submission_id` | UUID | FK → submissions (unique) |
| `extracted_text` | JSONB | What Gemini read from handwriting |
| `structural_analysis` | JSONB | Step-by-step method check |
| `feedback_items` | JSONB | Array of `{type, message}` objects |
| `suggested_marks` | INT | Gemini's mark suggestion |
| `model_used` | VARCHAR | e.g. `gemini-3.5-flash` |
| `generated_at` | TIMESTAMP | |

### `feedback_items` JSON structure
```json
[
  { "type": "correct", "message": "Chain rule applied correctly in step 2." },
  { "type": "error",   "message": "Learning rate term missing from gradient formula." },
  { "type": "warning", "message": "Diagram labels are unclear — ensure legibility." }
]
```

## 13. `sync_log` — Admin institute DB comparison history

One row per "Sync Now" run from the admin panel. Compares live
`students`/`faculty` records against the institute's mock DB records
(`mock/students.json`, `mock/faculty.json`) to catch drift.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `run_at` | TIMESTAMP | When this sync ran |
| `matched_count` | INT | Records identical between DB and institute records |
| `new_count` | INT | Institute records not yet onboarded (no account created yet) |
| `mismatch_count` | INT | Flagged discrepancies needing manual review |
| `mismatches` | JSONB | Array of `{category, identifier, name, details}` objects |

### `mismatches` JSON structure
```json
[
  {
    "category": "mismatch_student",
    "identifier": "EN2021001",
    "name": "Rahul Kumar",
    "details": "active status: true → false"
  }
]
```

## 14. `notifications` — In-app notifications

One row per notification per recipient. A single triggering event (e.g.
a new assignment posted to a section) creates one row per student in
that section, not one shared row.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `user_id` | UUID | FK → users |
| `title` | VARCHAR | Short heading, e.g. "Assignment Graded" |
| `message` | TEXT | Full notification text |
| `link_url` | VARCHAR | Optional — where clicking the notification navigates to |
| `is_read` | BOOLEAN | Defaults to `false`; the bell UI only ever shows unread notifications |
| `created_at` | TIMESTAMP | |

Triggered automatically in three places:
- Student notified when `evaluation.status` → `published` (graded)
- Faculty notified when `submission.status` → `ai_done` (ready to evaluate)
- Student notified when a new assignment is published (`status: active`) to their section

## Storage Strategy

| What | Where |
|---|---|
| Handwritten pages (images/PDFs) | Cloudinary (CDN), uploaded as authenticated/private resources |
| File references | Stored in `submissions.file_urls` as `resourceType\|format\|publicId` strings — converted to fresh signed URLs (1-hour expiry) on every read, never stored as a permanent public URL |
| All structured data | PostgreSQL via Supabase |
| Auth verification | Read-only sync from IPS Academy mock institute DB |
| Background job queue | Upstash Redis (BullMQ), triggered on a schedule via Upstash QStash |

CREATE TYPE submission_status AS ENUM (
  'submitted', 'ai_processing', 'ai_done', 'under_review', 'graded'
);

CREATE TYPE assignment_status AS ENUM ('draft', 'active', 'closed');

CREATE TYPE evaluation_status AS ENUM ('in_review', 'published');
